## Bottom line

The repository is ready for **incremental UI construction now**, even though it is not feature-complete: the author explicitly marks everything through **Phase 11D** as implemented, and Phase 11D exists specifically to let UI, CLI, network, replay, and AI teams build against one authoritative submission contract without UI-owned state mutation. ## What the current repository gives you right now

The new repository is not a generic rewrite; it is a **strict, engine-first reconstruction** with strong invariants. The README says all player choice must flow through `DecisionRequest` and `DecisionResult`, that UI/headless/network/replay/test drivers must all use the same decision path, and that the engine alone mutates authoritative state. The adapter contract reiterates that no adapter gets a private mutation path or a bypass around `DecisionRecord` / `EventRecord` generation. That is a very solid foundation for building multiple UIs without fragmenting game logic. ŅłĆciteŅłéturn5view0Ņłéturn5view1Ņłü

The roadmap is also clear about the current frontier. Everything through **Phase 11D** is treated as implemented, including the adapter scaffold, parameterized movement/placement proposal requests, Command phase body, objective control geometry, and Chapter Approved 2025ŌĆō26 mission pack data. At the same time, the same roadmap makes it clear that the visual UI itself is a later product slice, listed as **Phase 17C**, and server-authoritative network play as **Phase 17D**. That means the repository is intentionally set up so UI can begin **before** the full later gameplay stack is done. ŅłĆciteŅłéturn9view0Ņłéturn9view1Ņłéturn32view5Ņłü

The adapter layer already exposes the minimum primitives a first client needs. `LocalGameSession` can start a game, advance until there is a pending decision or terminal state, submit finite options or parameterized payloads, project a viewer-scoped game view, and fetch deterministic event deltas since a cursor. The projection already includes `battlefield_state`, `mission_setup`, `public_secondary_mission_choices`, `pending_decision`, `pending_proposal`, and `event_count`; the event stream already redacts hidden decision payloads and opponent-only secondary information when appropriate. The repositoryŌĆÖs Python packaging matters for framework choice. ## What the rules imply for the HUD and interaction model

The rules and roadmap point to a very specific style of UI. In the core rules, ranged attacks require that at least one model in the target unit is **visible** and **within the weaponŌĆÖs Range characteristic**; an enemy model is within range if its distance to the attacking model is less than or equal to the weaponŌĆÖs range. Units within Engagement Range generally are not eligible to shoot, with specific exceptions such as Pistols and some MONSTER/VEHICLE cases. That means your HUD should never treat ŌĆ£weapon rangeŌĆØ as a single yes-or-no ring; it needs to separate **raw distance**, **visibility**, and **current eligibility**. ŅłĆciteŅłéturn41view5Ņłéturn41view7Ņłéturn41view6Ņłü

Movement and charge reinforce the same point. Advancing is explicitly **M + D6** and causes a unit to lose the normal ability to shoot or charge that turn. Charging works only against targets within **12"**, does not require visibility, uses a **2D6** charge roll, and fails if the rolled distance is insufficient to reach Engagement Range of all declared targets while maintaining coherency. The UI implication is that a selected model or unit should expose at least three separate overlays: an **authoritative movement budget** overlay, a **charge declaration band** overlay, and a **charge-success preview** overlay that is not treated as deterministic until the engine phase and roll support exist. ŅłĆciteŅłéturn41view1Ņłéturn41view2Ņłéturn41view4Ņłü

The movement UI also needs to respect how physical movement is modeled. The core rules specify that when you move a model you can move it in a combination of straight lines and pivots, and no part of the base can move through an enemy model or off the battlefield edge. The repository roadmap mirrors that with a hard requirement for `PathWitness` and for movement legality to be validated via pathing and geometry rather than endpoint-only checks. So the first UI should not just drag a unit to a marker and hope the engine accepts it; instead it should record and visualize the actual planned path. ŅłĆciteŅłéturn14view3Ņłéturn5view0Ņłéturn5view1Ņłü

The mission pack also matters for networking and hidden information. The Chapter Approved 2025ŌĆō26 rules include **secret declaration steps** such as noting attached leaders, transports, and reserves before revealing them, and impose reserves restrictions such as ŌĆ£no more than half your units and no more than half your points in Reserves,ŌĆØ no arrival in battle round one, and destruction by the end of battle round three if required reserve units still have not arrived. The repositoryŌĆÖs viewer-scoped projections and event redaction line up with that kind of information asymmetry. Your network layer should therefore treat view projection as a **server-side security boundary**, not as a client courtesy. ŅłĆciteŅłéturn14view4Ņłéturn10view2Ņłéturn11view1Ņłü

The most important planning consequence is this: **movement, placement, and state inspection are the right first HUD targets**, because those are the areas Phase 11D was designed to expose. Fully authoritative shooting-range, charge-resolution, attack-allocation, and line-of-sight widgets belong to later roadmap slices, where the architecture introduces line-of-sight witnesses, cover policy, shooting declarations, attack pools, charge movement, and fight sequencing. You can still prototype those HUD affordances now, but some of them will be **preview/calculator-only** until later engine phases land. ŅłĆciteŅłéturn9view1Ņłéturn32view0Ņłéturn32view1Ņłéturn32view5Ņłü

## Network and protocol recommendation

The repository already tells you what the transport wants to look like. It wants a shared, authoritative submission path in which the engine emits a `DecisionRequest`, the adapter chooses a finite option or parameterized payload, the adapter converts that to a `DecisionResult`, the lifecycle validates it, and the engine records deterministic decision and event payloads. The contract also explicitly calls out viewer-scoped game views and event deltas. That maps very naturally onto a **server-authoritative WebSocket session** with four conceptual operations: `view_snapshot`, `events_since(cursor)`, `submit_option`, and `submit_payload`. ŅłĆciteŅłéturn5view1Ņłéturn10view2Ņłéturn11view1Ņłü

For the first implementation, I would use **JSON over WebSocket**, with Python-side schema validation through **msgspec**. The technical reason is simple: the repo already emphasizes **JSON-safe**, replay-facing payloads; the Python package already includes `orjson` and `msgspec`; PythonŌĆÖs async ecosystem has straightforward WebSocket support; and UnrealŌĆÖs C++ stack already has `IWebSocket`, `FHttpModule`, and `FJsonObjectConverter` for JSON-friendly client code. This gives you the shortest path to a Python client, a Python server, and an Unreal client all speaking the same contract. ŅłĆciteŅłéturn5view1Ņłéturn10view0Ņłéturn33search7Ņłéturn33search21Ņłéturn38view0Ņłéturn22search1Ņłéturn37view5Ņłü

One architectural adjustment is worth making immediately. Even though the current local convenience wrapper submits against the pending request implicitly, your **wire-level protocol should always include `request_id`** for every submission, because the contract is explicit that request IDs are what let network and replay clients detect stale drift early. This is a small addition with a big payoff in correctness. ŅłĆciteŅłéturn5view1Ņłéturn10view1Ņłü

On protobuf, FlatBuffers, and other generators, my assessment is:

| Option | Assessment for this project |
| --- | --- |

You do **not** need to introduce gRPC on day one to benefit from protobuf; plain protobuf-framed or HTTP/WebSocket-carried messages are also possible. ŅłĆciteŅłéturn25search0Ņłéturn25search4Ņłéturn25search7Ņłü |
| **FlatBuffers** | Powerful, especially for maximum memory efficiency and zero-copy access, and it was originally created at Google for game development and other performance-critical applications. ## Open questions and limitations

The biggest product limitation is roadmap timing: in the architecture document, the **shooting**, **charge**, and **fight** systems are later phases, while the visual UI itself is also a later milestone. That does not block UI work, but it does mean some of the more ambitious HUD ideas you namedŌĆöparticularly fully authoritative attack-threat and charge-threat displaysŌĆöshould be treated as staged work, not as day-one engine-backed truth. The evidence I found is much stronger for Interchange, glTF, Datasmith, and broader DCC-oriented pipelines. That is why my recommendation for Unreal is to **ingest STL if necessary, but normalize to glTF/GLB or another richer runtime format** before the asset hits the main game client. This conclusion is based on the sources reviewed plus engineering inference, and it is the place where a small proof-of-concept conversion spike would be most valuable.
